# Portfolio: Runmate (런메이트)

> **마라톤 일정 관리 및 러닝 데이터 분석 플랫폼**

Runmate는 흩어져 있는 마라톤 대회 정보를 통합하고, 사용자의 러닝 기록(TCX, CSV)을 시각화 및 분석하여 체계적인 훈련을 돕는 웹 서비스입니다. 단순히 기록을 저장하는 것을 넘어, 성취감을 자극하는 배지 시스템과 데이터 기반의 통계를 제공합니다.

---

## 🛠 Tech Stack

- **Framework**: Next.js 14 (App Router), TypeScript
- **Database**: PostgreSQL (Supabase), Prisma ORM
- **Auth**: NextAuth.js (Google/Kakao/Naver Social Login)
- **Visualization**: Chart.js, React-Leaflet (Map)
- **Data Parsing**: fast-xml-parser (TCX), iconv-lite (EUC-KR Crawling)
- **Styling**: TailwindCSS

---

## 🚀 Key Features

### 1. 지능형 마라톤 일정 크롤링 & 동기화

- **비정형 데이터 구조화**: 국내 주요 마라톤 일정 사이트(`roadrun.co.kr`)를 크롤링하여 파싱합니다.
- **인코딩 대응**: 레거시 사이트의 `EUC-KR` 인코딩을 `iconv-lite`로 처리하여 데이터 깨짐 없이 수집합니다.
- **자동 분류**: 정규표현식을 통해 대회명에서 코스(Full, Half, 10K)를 자동 추출하고, 주소 데이터를 기반으로 광역시도 단위의 필터링을 지원합니다.

### 2. 피트니스 데이터 임포트 및 GPS 시각화

- **멀티 포맷 지원**: Garmin, Strava 등에서 사용하는 표준 `TCX(XML)` 파일과 커스텀 `CSV` 파일을 지원합니다.
- **GPS 데이터 최적화**: 수천 개의 GPS 좌표 데이터를 `RDP(Ramer-Douglas-Peucker)` 유사 알고리즘으로 샘플링하여, 지도 렌더링 성능을 최적화하고 브라우저 부하를 줄였습니다.
- **자동 페이스 계산**: 이동 거리와 시간을 바탕으로 구간별/전체 페이스를 자동 계산하여 시각화합니다.

### 3. 데이터 기반 통계 대시보드

- **활동 히트맵**: GitHub 잔디 스타일의 52주 활동 히트맵을 구현하여 사용자의 꾸준함을 시각화했습니다.
- **비교 분석 그래프**: 작년 대비 올해의 월별 누적 거리 비교, 페이스 추이(역축 그래프) 등을 통해 성장을 체계적으로 추적합니다.
- **개인 기록(PB) 관리**: 최장 거리, 최고 페이스, 최장 시간 기록을 자동으로 업데이트하고 관리합니다.

### 4. 게이미피케이션 기반 배지 시스템

- **성취 로직**: 월간 마일스톤(50km, 100km 등), 연속 러닝(Streak), 개인 기록 경신 시 실시간으로 배지를 수여합니다.
- **중복 방지**: 신규 배지 획득 여부를 `findUnique` 로 먼저 확인한 뒤 `create`하는 패턴으로, 활동 기록 수정·삭제 시에도 이미 수여된 배지가 재발급되지 않도록 처리합니다.
- **Fire-and-forget sync**: 배지 집계 로직을 `await` 없이 백그라운드에서 실행하여 API 응답 속도에 영향을 주지 않습니다.

### 5. 실시간 알림 시스템

- **D-Day 알림 자동 생성**: `GET /api/notifications` 호출 시 참가 신청된 대회를 조회하여, 대회일 7일·1일 전에 해당하는 알림을 서버에서 자동 upsert합니다. `(userId, type, refId)` 유니크 제약으로 중복 생성을 원천 차단합니다.
- **배지 획득 알림**: 신규 배지 수여 시 알림 레코드를 함께 생성하여 획득 시점을 사용자에게 즉시 전달합니다.
- **TopBar 통합**: 미읽음 알림 수를 벨 아이콘에 뱃지로 표시하고, 드롭다운 패널에서 상대 시간(`방금 / n분 전 / n시간 전`)과 알림 유형별 아이콘·색상으로 구분해 보여줍니다. 패널을 열면 전체 읽음 처리가 자동으로 실행됩니다.
- **폴링 전략**: TanStack Query의 `refetchInterval`을 5분으로 설정하여, 서버 부하 없이 준실시간으로 알림을 갱신합니다.

## 🔍 Technical Challenges & Solutions

### Issue 1: 대용량 GPS 데이터로 인한 지도 렌더링 성능 저하

- **Problem**: 1초 단위로 기록된 TCX 파일의 좌표(수천 개)를 그대로 지도에 뿌릴 경우 브라우저 메모리 점유율이 급증하고 인터랙션이 끊기는 현상이 발생함.
- **Solution**: 서버 사이드 파싱 단계에서 좌표 데이터의 밀도를 분석하여, 전체적인 경로의 모양은 유지하되 포인트 개수를 약 400개 내외로 압축하는 **좌표 샘플링 로직**을 도입하여 렌더링 속도를 80% 이상 개선함.

### Issue 2: CSV 파일 포맷의 다양성 대응

- **Problem**: 플랫폼마다 CSV 헤더 명칭과 구조가 달라 범용적인 임포트가 어려웠음.
- **Solution**: 헤더 이름의 유사도를 판단하는 매핑 로직과, 특정 플랫폼(Garmin 등)의 특수한 'Summary' 행을 감지하는 **지능형 포맷 감지 알고리즘**을 구현하여 사용자 편의성을 높임.

### Issue 3: 페이스(Pace) 데이터의 특수성 시각화

- **Problem**: 페이스는 '낮을수록 좋은' 기록이므로 일반적인 선형 그래프로는 직관적인 이해가 어려움.
- **Solution**: Chart.js의 **Y축 역전(Reverse Scale)** 옵션을 활용하고, `mm'ss"` 포맷의 시간 데이터를 초 단위로 변환하여 처리하는 커스텀 틱(Tick) 로직을 적용하여 직관적인 성적 향상도를 표현함.

### Issue 4: 알림 중복 생성 방지 및 D-Day 계산

- **Problem**: 알림을 매 API 호출 시 생성하는 구조상, 새로고침할 때마다 D-Day 알림이 중복으로 쌓일 수 있음. 또한 크롤링으로 수집한 마라톤 날짜가 `YYYY-MM-DD`, `YYYY년 M월 D일`, `YYYY.MM.DD` 등 비정형 문자열로 혼재해 날짜 연산이 불가능함.
- **Solution**: `(userId, type, refId)` 유니크 제약 + `upsert`로 중복을 DB 레벨에서 원천 차단하고, `refId`에 대회 날짜를 포함시켜 동일 대회의 재신청 후에도 알림이 안전하게 재생성되도록 설계함. 날짜 파싱은 정규표현식 3종(`ISO / 한국어 / 점 구분`)을 순차 매칭하는 `parseDateString` 함수로 처리함.

## 🗄 Database Schema Design (Highlights)

- **Users & Participations**: N:M 관계를 통한 대회 신청 관리.
- **Activities & RouteData**: GPS 데이터를 효율적으로 저장하기 위한 JSON 타입 활용.
- **BadgeDefinitions & UserBadges**: 배지 획득 조건을 정규화하여 관리하고, 월별 획득 여부를 유니크 인덱스로 제어하여 중복 수여 방지.
- **Notifications**: `(userId, type, refId)` 복합 유니크 제약으로 D-Day·배지 알림의 중복을 DB 레벨에서 차단. `refId`는 `marathon:{id}:{date}` / `badge:{code}:{year}:{month}` 형식으로 자연키처럼 활용.

## 🎨 Layout Preview

- **Main**: 다가오는 대회 일정과 지역·종목 필터 + 페이지네이션.
- **Activity**: 지도를 포함한 상세 러닝 분석 드로어 + 활동 테이블.
- **Stats**: 52주 활동 히트맵, 월별 비교 그래프, 개인 최고기록(PB), 배지 모음.
- **TopBar**: 알림 벨(미읽음 뱃지) + 드롭다운(D-Day·배지 알림 목록) / 프로필 드롭다운(이번 달 km · 로그아웃 · 회원 탈퇴).
